# Updating the Website

The website at `website/` uses auto-generated data extracted from the Rust source code. The file `website/src/data/sourceData.ts` is generated by `website/scripts/extract-source-data.mjs` and must stay in sync with the codebase. It gets stale when CLI commands, patterns, workflow model, or version change without regenerating the website source data.

## Tracking Mechanism

The file `.claude/skills/update-website/.last-updated` contains the git commit hash from the last time the website was comprehensively updated. Use this as the baseline for discovering what changed.

## Discovery Process

1. Read the baseline commit hash:
   ```sh
   BASELINE=$(cat .claude/skills/update-website/.last-updated)
   ```

2. List all commits since the baseline:
   ```sh
   git log --oneline "$BASELINE"..HEAD
   ```

3. Check what files changed:
   ```sh
   git diff --name-only "$BASELINE"..HEAD
   ```

4. Categorize the changes using the source mapping below to determine if the website needs updating.

5. Read the affected source-of-truth files and the current `website/src/data/sourceData.ts`. Fix any discrepancies.

## Source Mapping

| Changed files / commit scope | What to update |
|------------------------------|---------------|
| `zig-cli/Cargo.toml` (version) | Regenerate source data (version badge) |
| `zig-cli/src/cli.rs` (Command enum) | Regenerate source data (commands list) |
| `zig-cli/src/cli.rs` (WorkflowCommand enum) | Regenerate source data (workflow subcommands) |
| `zig-cli/src/cli.rs` (Pattern enum) | Regenerate source data (patterns), update `Patterns.tsx` |
| `zig-core/src/workflow/model.rs` (Step, VarType) | Regenerate source data (step fields, variable types) |
| `manpages/*.md` | Regenerate source data (manpage topic list) |
| New commands or patterns | Terminal demos, feature descriptions, pattern examples |

## Implementation Files

### Primary

- `website/src/data/sourceData.ts` — Auto-generated from Rust source (do not edit manually)

### Secondary (read-only, sources of truth)

| Source of truth | What it tells you |
|----------------|-------------------|
| `zig-cli/Cargo.toml` | Version number |
| `zig-cli/src/cli.rs` | All CLI commands, workflow subcommands, patterns |
| `zig-core/src/workflow/model.rs` | Step fields, variable types |
| `manpages/*.md` | Manpage topic list |

### Extraction script

- `website/scripts/extract-source-data.mjs` — The extraction logic

### Website components that consume source data

| Data field | Used by |
|-----------|---------|
| `version` | `Hero.tsx` (version badge) |
| `commands` | `Features.tsx` (count), `GettingStarted.tsx` (reference) |
| `workflowSubcommands` | `GettingStarted.tsx` (reference) |
| `patterns` | `Features.tsx` (count), `Patterns.tsx` (cards) |
| `stepFields` | `Features.tsx` (count), `WorkflowFormat.tsx` (reference table) |
| `varTypes` | `WorkflowFormat.tsx` (description) |

### Editorial content (not auto-generated, may need manual updates)

- `website/src/components/HowItWorks.tsx` — Three-step flow
- `website/src/components/BuiltOnZag.tsx` — Architecture diagram and comparison
- `website/src/data/terminalDemos.ts` — Terminal animation sequences

## Update Checklist

- [ ] Read baseline from `.last-updated` and run `git log` to identify changes
- [ ] Read source-of-truth files for affected areas
- [ ] Regenerate source data: `cd website && npm run extract`
- [ ] Verify `sourceData.ts` version matches `zig-cli/Cargo.toml`
- [ ] Verify all commands from `Command` enum are present
- [ ] Verify all workflow subcommands from `WorkflowCommand` enum are present
- [ ] Verify all patterns from `Pattern` enum are present
- [ ] Verify step fields match the `Step` struct
- [ ] Verify variable types match the `VarType` enum
- [ ] If new commands or patterns were added, update editorial content:
  - [ ] Terminal demo sequences in `website/src/data/terminalDemos.ts`
  - [ ] Feature descriptions in `website/src/components/Features.tsx`
  - [ ] Pattern examples in `website/src/components/Patterns.tsx`
- [ ] Build the website: `cd website && npm run build` — must succeed
- [ ] Consider whether `update-readme` and `update-manpages` skills should also be run
- [ ] Update `.claude/skills/update-website/.last-updated` with current HEAD commit hash:
  ```sh
  git rev-parse HEAD > .claude/skills/update-website/.last-updated
  ```

## Verification

1. Run `cd website && npm run build` — must succeed
2. Verify `sourceData.ts` version matches `zig-cli/Cargo.toml`
3. Verify command count matches the `Command` enum variants
4. Verify pattern count matches the `Pattern` enum variants
5. Ensure no components were accidentally broken
6. Confirm `.last-updated` file was updated

## Skill Self-Improvement

After completing an update session, improve this skill file:

1. **Add new source files**: If new Rust source files provide data for the website, add them to the extraction script and this mapping.
2. **Add new components**: If new website components consume source data, add them to the component mapping.
3. **Record patterns**: If you found a recurring update pattern not documented here, add it.
4. **Commit the skill update** along with the website updates so improvements are preserved.
